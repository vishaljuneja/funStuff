          I wrote a small class mirrored around the DirectedCycle class provided on the booksite. Its job is simply to check if a given digraph is both acyclic and has only one root. It completes a DFS traversal of the graph only if it checks out since I can always return early out of the check once I find either a cycle or a second root.

          As I wrote earlier in another thread, a root is simply a vertex with no outgoing edges, so it is easy to search for them.

          Mostly I did things suggested in the hints:

          1. Created DeluxeBFS to provide a list of ancestors. Then I check v's list vs. w's list and not V x V.

          3. DeluxeBFS constructors are called only twice in the lifetime of one SAP. DeluxeBFS supports change_source_to(new_v) to patch up the marked[] and distTo[] arrays based on the cached ancestors of the previous v instead of initializing them from scratch.

          I feel strange calling my class DeluxeBFS though when I had to strip down unnecessary parts like the edgeTo[] array to make room for storing two hash maps (not very deluxe anymore without being able to compute pathTo(), haha). My SAP class has one persistent DeluxeBFS object, which stores

          - A set of vertices to reset when the object has to be reused
          - the marked[] and distTo[] arrays
          - A hash map to map ancestors to their distTo[] value from the given source vertex for client use
          - A set source vertex/vertices operation that resets the set of vertices mentioned above and then runs dfs from the source(s)

          These hash maps cost me a good chunk of memory though. I hate having my memory footprint 5% greater than the reference for the SAP object. I should read up a bit to see if the algs4 linear probing or separate chaining hash ST's look like they'd take less memory than the java.util.HashMaps I'm using currently.

          For instance, when I need to compute length(v, w) after a cache miss I need to clone() the hashmap for source vertex v (giving the distance from the source vertex v to each ancestor vertex) before setting the source vertex to w and thus invalidating the distTo[] values for v. Then for w I can just the DeluxeBFS's hashmap since the DeluxeBFS for w is still valid. Maybe it can be done better?

          Has anyone gotten good results implementing a larger cache than just the previous computation? (that's what I'm using in mine)
