    First, notice that you are allocated by the grading script substantially more space than needed. Based on the data structures introduced in the first set of lectures, is there any possible way you can employ more space to eliminate backwash.

    Of course, to answer this you would need to know what backwash is:

    Now, the visualizer fills in a square if it isFull() – you need to consider how your algorithm checks if something "is full". To start you off, remember that for efficiency's sake, you are using a "virtual" top and bottom node to check if there is a percolation throughout. However, after it percolates how does the method work?

    You can physically think about backwash as water that pours through the diagram that is then injected from the bottom. Which squares, then, would be filled?

    This problem was particularly interesting because efficient outcomes ranged in size by a factor of almost 9, the best had a coefficient of 2, the worst 16. There are some remarkable posts and discussions that took place, which we will work on uploading eventually, that discuss the intricacies of the problem.

      Note that the problem that percolates() and isFull() are trying to solve is different:

      For percolates() we want to know if a 'hidden node' at the top and a 'hidden node' at the bottom are connected.
      For isFull() we only want to know if a given node is connected to the top 'hidden node'

      A simple solution would be to maintain 2 data structures - one which has a hidden top and bottom and one which has a hidden top only. In the interest of promoting discussion I'll not say exactly what these could be would be used for and how this solves the backwash problem.

        • Vinicius Kwiecien Ruoso· 3 months ago 

          I've solved the backwash problem using two approaches.

          The first is the one already mentioned in this thread, where you have two structures, one to control if the system percolates and another to control if a given cell is full. My code with that approach passes the memory limits and the N=1024 case runs in 0.5s. The memory usage is increased only by ~ 2N2∗4 bytes.

          The second approach I implemented have a memory overhead of ~ N2∗1 bytes. The idea is to keep track of cells that are "full", so when opening a new cell you can check if some neighbour is already full and update your structure. As this implementation touches at most in N2 elements, and checking if a cell is full takes constant time, it is faster than the first one. For the N=1024 test it takes 0.31s to complete.

          • Vinicius Kwiecien Ruoso· 3 months ago 

            It's not. I don't check all N2 elements each call. If I end up calling union in some moment, it's because I have an opened cell connected to the current cell. If that cell that I just looked at is already full, I must say that the current cell is full. That's the easy part. Now you have to make sure that if you have connected with an "empty" cell, you also have to go there and set it as "full". If that cell have some open neighbour that is not "full", you should keep going, until you set everyone that is connected to that current node as "full".

            This may look like it's quadratic, but it will cost one operation to set a cell as full, and it will set a cell as full only once. So it can touch all N2 cells in one run, but once it has been done, all calls will take constant time.

              • Brian Reischl· 3 months ago 

                I think that disconnecting the virtual bottom node would either not help or cause bugs, depending on how you initially attach the bottom virtual node to the bottom row nodes.

                When you start off you have to connect all the nodes in the bottom row to the virtual bottom. Depending on which order you pass the arguments to .union(), you'll either end up with the bottom virtual node being the root of that subtree, or one of the bottom row nodes being the root of the subtree.

                If one of the bottom row nodes is the root of the subtree, then detaching the virtual bottom won't make any difference because it'll be a leaf node.

                If the virtual bottom node is the root, then it will be the root of a largish (N+1 nodes) subtree. As you open more nodes near the bottom, they'll most likely become subtrees of the virtual bottom node. When you disconnect it, you'll also disconnect that entire subtree. That would cause all those other nodes to erroneously report being not full.

                Just found following suggestion to fix the backwash problem (I am not sure it's right to post it here):

                Blockquote One last technical comment about percolation, in response to a question asked by Clayton Raithel: if one uses the suggested trick of adding dummy open rows at the top and bottom to make the test for percolation more efficient, one encounters the “backwash” problem mentioned in the checklist for the assignment: when percolation occurs, all components that contain an open cell in the bottom row are declared “full” (and will be visualized as such), because they are connected through the dummy bottom row to an actual full cell in the bottom row. This is a symptom of a possibly more serious problem: the implementation does not actually keep track of all the connected components, since components with cells in the top row are combined, as are components with cells in the bottom row. This means that one cannot keep track of additional statistic, such as the number of components, or the size of the largest component. Here is what I think is a better and more disciplined way to keep track of whether the system percolates. Store two bits with each component root. One of these bits is true if the component contains a cell in the top row. The other bit is true if the component contains a cell in the bottom row. Updating the bits after a union takes constant time. Percolation occurs when the component containing a newly opened cell has both bits true, after the unions that result from the cell becoming open. Unfortunately one must extend the union-find API to support such bits (or any other information associated with a set). For this and other reasons, I prefer the following alternative API: Each set has a unique canonical element, which the implementation gets to choose. The find method has only one argument, p, and it returns the canonical element of the set containing p. The union method has two arguments, both canonical elements. It combines the sets containing these canonical elements and chooses a canonical element for the new set. One also needs a method to initialize the data structure. With such an API, one can associate arbitrary information about the sets with their canonical elements. Blockquote
